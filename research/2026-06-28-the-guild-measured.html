Report #3 · 2026-06-28

The guild, measured: three specialists, three lessons

All three Go specialists — dev, test, review — are now trained (locally, $0) and measured on objective, hidden-signal benchmarks. The result isn't a clean sweep, and we wouldn't publish it if it were dishonest: one clear win, one marginal win, and one result that taught us more by losing than a win would have.

Lesson 1 — specialization is strongest on disciplined tasks

The go-test model is the cleanest win (+2, see Report #2). Test-writing is a discipline — cover the edges, catch the bug — and the base isn't saturated there, so targeted training moves the needle. go-dev wins only marginally over its base, because the code base is already a Go powerhouse from pretraining; the bigger gap (+3) is against a general model, which is the real GuildLM claim (see Report #1).

Lesson 2 — measure every specialist on its own job

go-review scored below its base on the edit benchmark — and that result is the most useful of the three, because it exposed our own mistake. The edit benchmark asks for a clean corrected file. But go-review was trained to write reviews: find the bug, explain why it's wrong, then show the fix. Optimizing for explanation traded away some clean-code-emission, so on a pure-edit metric it underperforms.

The benchmark was wrong, not the model. Editing is not reviewing. A review specialist must be judged on whether it correctly identifies the bug — which needs a dedicated review benchmark we haven't built yet. Reporting the mismatch is the honest move; quietly swapping in a flattering number is not.

Lesson 3 — the value of a specialist is task-dependent

Put the three together and the shape of the bet is clear: a small specialist's edge is largest on focused, disciplined work (testing, reviewing) and smallest on raw generation where a strong code base already shines. The guild isn't "train a model per role and assume each wins" — it's "build specialists where specialization pays, and measure each on the job it actually does."

What's next

  • Build an objective go-review benchmark — buggy code in, and score whether the review names the real defect — so go-review gets a fair test.
  • Close the shared gaps every model still misses (Unicode-rune handling, JSON round-trips).
  • Wire the measured specialists into the agent loop (the GuildLM "builder"), where dev writes, test tests, and review checks.

Every run, win or loss, reported here as it happens.
